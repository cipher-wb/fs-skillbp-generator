# 关于构建“基于自然语言的 AI 技能蓝图自动生成管线”的可行性与技术改造提案

**To: 客户端/框架开发团队**
**From: 策划团队 & AI 辅助构建系统**

## 〇、 愿景与背景
目前项目的技能/AI 编辑器（基于 `NodeEditor` 和 `NodeGraphProcessor`）已经具备了极高的工业化成熟度。为了进一步解放生产力，策划团队希望打通一套 **“AIGC 辅助配表与连线”** 的管线：
> **核心目标**：策划通过自然语言描述技能逻辑（如：“前摇10帧，发射追踪子弹，命中附加击飞Buff”），AI 直接理解意图并生成能在 Unity 编辑器中无缝打开、且连线和参数均已配置正确的技能图谱蓝图。

经过对目前底层 JSON 序列化文件结构（如 `Jsons/宗门技能/...`）的深度解析与逆向验证，**该方案在架构上是高度可行的**。当前的纯数据驱动和拓扑图结构非常契合 AI 解析。但在完全自动化的过程中，AI 面临一些底层序列化机制带来的硬性阻碍，需要程序团队提供一定的技术协助和接口支持。

---

## 一、 AI 直接生成底层 JSON 的技术痛点与风险

在尝试让 AI 越过 Unity 引擎，直接生成并写入 `Saves/Jsons/` 目录下的 `.json` 文件时，遇到了以下机制上的强校验壁垒：

1. **`TableTash` 强校验墙**
   - **痛点**：每一个被序列化的 `ConfigBaseNode` 都带有一个 `TableTash` 字段（如 `"0C2C5D4A7DC9280429315C2EEAD44FDF"`），用于检测 C# 数据结构变更。
   - **风险**：AI 无法实时计算或获取最新代码的哈希值。一旦程序侧修改了节点数据结构，AI 生成的零值哈希（`00000000...`）或旧哈希会导致编辑器加载时丢弃该节点数据或触发容错报错。

2. **魔法数字与业务逻辑“黑盒” (Magic Numbers)**
   - **痛点**：序列化后的 `ConfigJson` 拍平了大量枚举值。例如 `SkillEffectType: 14`（可能代表加Buff）、`ParamType: 1, Value: 91`（可能代表读取施法者属性）。
   - **风险**：缺乏项目专属的“数据字典”，AI 只能靠猜测连线逻辑，极易产生不存在的枚举值（幻觉），导致游戏 Runtime 报错或逻辑瘫痪。

3. **全局 ID 与 GUID 分配冲突**
   - **痛点**：技能 ID、`Config2ID` 以及连线 `edges` 所需的 `System.Guid` 必须在全局严格唯一。
   - **风险**：AI 独立生成的 ID 极易与本地已有配置（如海量的宗门技能）或 SVN 上的文件产生主键冲突。

## 二、 落地实施方案与程序支持诉求

为了让这套自动化管线真正健壮、无痛地落地，强烈建议放弃“让 AI 直接暴力生成复杂图谱 JSON”的思路，转而采用**“AI 语义解析输出 DSL -> 编辑器引擎反向构建”**的策略。

需要程序团队评估并支持以下两个阶段的方案：

### 方案 A（短期低成本）：提供《AI 专用数据字典库》
若近期不打算修改编辑器代码，我们需要程序定期导出一份（或维护一份）机器可读的映射字典（JSON/YAML 格式即可），赋能 AI 大脑：
1. **节点功能与类型映射**：如 `1=顺序执行 (TSET_ORDER_EXECUTE)`，`8=创建子弹 (TSET_CREATE_BULLET)`。
2. **标签与属性枚举字典**：如 TargetType 枚举（`91=施法者`，`59=X坐标`等）。
3. **特效/资源路径白名单**：常用光效、子弹模型的规范化路径列表。
*注：此方案下，AI 仍需手动处理复杂的 GUID 连线，且受制于 `TableTash` 的更新。*

### 方案 B（长期最优解）：开发“AI 指令集 (DSL) 导入器”
这是最符合工业化标准的设计。通过定义一套极简的“中间态描述语言”，将脏活累活（排版、打哈希、算 UUID）交还给 C# 框架处理。

**1. 策划/AI 交互流：**
策划输入自然语言 -> AI 生成极简且高维的 YAML/JSON 配置树（DSL）。
```yaml
# AI 生成的 DSL 示例 (极简、无 GUID、无哈希)
Skill_90001:
  Name: "艾希W"
  BaseInfo: { CastFrame: 10, CdTime: 60, Cost: 60 }
  LogicGraph:
    - Parallel: # 顺序分发节点
      - WarningLine: { Length: 1000, Width: 150 }
      - FireBullet: 
          Speed: 1000
          Model: "Effect/14003_skill4_fly"
          OnHit: AddBuff(BuffId: 10002) # 直接通过语义关联
```

**2. 程序接口支持：**
在现有的 `SkillGraphWindow` 顶部工具栏增加一个 **[从 AI 脚本导入 (Import from DSL)]** 按钮。
- **底层工作流**：读取该 YAML 文件，利用编辑器已有的 API 在内存中实例化节点（`new TSET_CREATE_BULLET()`），自动计算并在 UI 上排版（Position），调用底层的 `Connect` 接口建立 `edges`，并利用系统的 `GenerateID()` 分配唯一 ID。
- **优势**：
  - 完美绕过了 `TableTash` 校验，生成的必定是带有当前版本最新哈希的合法节点。
  - C# 层的创建逻辑自动规避了必填字段的遗漏，连线逻辑受到强类型保护。
  - AI 的输出 Token 数量极大缩减，降低了大语言模型的出错率，响应速度提升 10 倍。

## 三、 总结评估

让 AI 深入参与 Node-Based 技能逻辑构建是下一代研发管线的趋势。我们目前的底层数据分离做得很好，为接入 AI 铺平了道路。

如果程序侧能够提供 **方案 B (DSL 导入解析器)** 的支持，策划团队即可彻底摆脱繁琐的手动建节点与填参过程，实现“一句话生成技能骨架”，再由策划进行人工微调的极速迭代流。希望程序团队能对上述接口的开发成本和接入可行性进行评估。
