# 测试蓝图深入解析报告
**文件**: `SkillGraph_10121009_【测试】吴波技能测试蓝图.json`

正如其文件名**“测试蓝图”**所示，这并不是一个单一的、线性的常规技能，而是一个策划（吴波）用于**沙盒测试**多种独立战斗机制的复合蓝图。整个图谱由一个挂载了实际执行逻辑的主技能节点，以及若干个断开连接（但内部逻辑闭环）的测试功能块组成。

以下是极其详细的内容与运作逻辑分析报告：

## 一、 核心骨架：主技能逻辑流
在图谱中，真正与技能入口 `SkillConfigNode (ID: 320008)` 连通并会被主流程触发的只有一条极为简短的线：
*   **触发源**：`SkillConfigNode` (ID: 320008)。配置了基础信息如 `CdTime: 210`（21秒冷却，假设10帧=1秒）、`SkillBaseDuration: 45`（技能总时长）、`SkillRange: 700`（射程）。
*   **连线逻辑**：技能的 `SkillEffectExecuteInfo.SkillEffectConfigID` 端口连接到了节点 `32003758`。
*   **执行结果**：`TSET_CREATE_EFFECT` (ID: 32003758) 节点。这是一个**“创建特效”**的执行节点，它又连接到了 `ModelConfigNode` (ID: 3200494)，即加载并播放路径为 `Effect/GW20149_skill3_circle` 的光圈模型特效。
*   **总结**：在游戏里真正按下这个技能时，它仅仅会在原地播放一个光圈特效，没有伤害判定。这是一个最基础的“播放表现”测试。

---

## 二、 极度硬核：子弹与随机弹道计算逻辑（未连入主流程）
这是蓝图中最复杂、最能体现节点编辑器工业化能力的一个独立模块。策划在这里用数学节点搭建了一个**“每次发射，落点随机偏移的子弹”**逻辑：

### 1. 子弹发射器
*   `TSET_CREATE_BULLET` (ID: 32004213)：创建子弹节点。
*   `BulletConfigNode` (ID: 320205)：子弹的具体配置。飞行速度 `Speed: 300`，挂载了模型 `Effect/GW20180_skill5_fly`。
*   **轨迹核心**：它的 `FlyType`（飞行类型）被设为了 5（可能是某种自定义轨迹或贝塞尔曲线），并且其 `TracePathParams`（寻路参数）引用了两个外部逻辑节点 `32004227` 和 `32004228`。

### 2. 随机落点 X/Y 坐标解算过程
这两条对称的逻辑链展示了节点间数据流的传递：
*   **X坐标计算链**：
    *   首先通过 `TSET_GET_ENTITY_ATTR_VALUE` 节点获取施法者的基础坐标 X (属性 59) 和坐标 Y (属性 60)。
    *   通过 `TSET_CREATE_RANDOM_NUM` 生成一个 `0~60` 的随机角度，和一个 `100~800` 的随机距离。
    *   这四个值作为参数（Input）流入 `TSET_GET_POS_X_BY_ANGLE_DISTANCE` (ID: 32004217) 节点。该节点底层封装了三角函数，计算出：`目标X = 基础X + cos(随机角度) * 随机距离`。
    *   最后，计算结果流入 `TSET_MODIFY_SKILL_TAG_VALUE` (ID: 32004227) 节点，**将算出的 X 坐标强行写入到 技能变量标签(Tag) `1001` 中**。
*   **Y坐标计算链**：
    *   与上述逻辑完全对称，调用 `TSET_GET_POS_Y_BY_ANGLE_DISTANCE` 计算出目标的 Y 坐标。
    *   将 Y 坐标写入到 **技能变量标签(Tag) `1002` 中** (ID: 32004228)。

### 3. 模块运作逻辑
当子弹尝试飞向目标时，子弹底层逻辑会调用并执行 `32004227` 和 `32004228`，实时算出带随机偏移的 X(1001) 和 Y(1002) 坐标，子弹进而向这两个变量指定的坐标飞行，实现每次释放轨迹都不一样的“散弹”或“流星雨”效果。

## 四、 底层数据绑定的印证（从架构层面看）
从这个 JSON 的细节中，印证了我们之前代码分析中提到的管线设计：

1.  **强数据弱连线**：图谱的边(`edges`) 记录了 `PackedParamsOutput` 和 `InputPort: 0`。但在 JSON 的 `ConfigJson` 中，连线关系已经被直接拍平成了数组。例如，节点 `32004217` 的 Params 里直接存了上游节点的 ID：`[{"Value":32004215,"ParamType":2...}]`。`ParamType: 2` 告诉底层这是一个“引用ID”，而不是常数。这说明**连线在保存的一瞬间，已经完全转化为了配置表所需的外键 ID**。
2.  **TableTash 校验码机制**：每个节点都带有如 `"TableTash": "0CFA05568A66FEA1DF3BA6FE40DB7080"` 这样的哈希值。这证明如果某天程序偷偷修改了 `TSET_CREATE_EFFECT` 的 C# 结构，这个哈希值对不上，框架就能在下次打开这个图谱时进行兼容性修复。
3.  **坐标完全解耦**：JSON 将逻辑数据(`data`) 和 视觉排版数据(`position: x,y,width,height`) 完全分开。这使得最终导表给游戏客户端用时，可以轻易地剔除掉坐标、颜色等 Editor-Only 数据，保证极小的包体。
